START PROMPT

Jesteś senior full-stack developerem, architektem aplikacji SaaS/LMS, security engineerem i product designerem UI/UX. Twoim zadaniem jest doprowadzić istniejącą aplikację edukacyjną AI do stanu 100% komercyjnego MVP gotowego do produkcyjnego odbioru.

Pracujesz na istniejącym repozytorium platformy edukacyjnej. Masz bezwzględnie oprzeć się na załączonym audycie technicznym:
Audyt_Techniczny_Kodu_MVP_Platforma_Edukacyjna_AI_v2.docx

Najpierw przeczytaj cały audyt, potem przejrzyj kod repozytorium. Nie zgaduj. Nie rób powierzchownych zmian. Nie zostawiaj TODO zamiast działającego kodu. Nie oznaczaj zadania jako ukończone, dopóki aplikacja nie przejdzie builda, typechecka, testów, migracji, Docker run i checklisty MVP.

CEL GŁÓWNY:
Zrobić z obecnego MVP pełną, profesjonalną, komercyjną platformę edukacyjną z modułem AI, wyglądającą i działającą jak nowoczesny portal LMS klasy premium. Aplikacja ma być gotowa do sprzedaży, użycia przez prawdziwych uczniów i wdrożenia na VPS.

STACK ISTNIEJĄCY:
- Frontend: React, TypeScript, Vite, Tailwind, shadcn/ui lub istniejące UI primitives.
- Backend: Express, TypeScript.
- Baza: PostgreSQL + Drizzle.
- Auth: poprawić do produkcyjnego standardu.
- AI: Gemini po stronie backendu.
- Wideo: Bunny.net video ID / videoUrl embed.
- Płatności: Przelewy24.
- Rysowanie/zadania: Excalidraw.
- Docker/VPS: Docker Compose, nginx/web, API, PostgreSQL.
- Testy: Vitest/backend + Playwright/e2e + typecheck/build.

ZASADA NADRZĘDNA:
MVP ma być spełnione w 100%. Rozszerzenia poza MVP, takie jak fiszki, kody promocyjne, CRM, dodatkowe formularze itp., nie są wymagane. Jeśli istnieją i przeszkadzają produkcji, ukryj je za feature flagą albo odseparuj od wymaganych env. Priorytetem jest kompletne, stabilne MVP.

2. BLOKERY P0 — MUSZĄ BYĆ NAPRAWIONE
====================================================

P0-01: Publiczny endpoint listy tematów sekcji
Problem:
GET /sections/:sectionId/topics jest publiczny albo niedostatecznie chroniony. Nie może zwracać tematów lekcji użytkownikowi bez dostępu.

Wymagania:
1. Dodaj helper w access.ts:
   getCourseIdBySectionId(sectionId: number): Promise<number | null>
2. Endpoint /sections/:sectionId/topics musi mieć:
   - requireAuth,
   - requireCourseAccess,
   - walidację sectionId jako positive int,
   - 400 dla błędnego ID,
   - 404 dla nieistniejącej sekcji,
   - 403 dla usera bez dostępu,
   - admin ma dostęp.
3. Dodaj testy:
   - niezalogowany dostaje 401,
   - zalogowany bez accessGranta dostaje 403,
   - user z dostępem dostaje listę,
   - admin dostaje listę,
   - invalid sectionId daje 400,
   - brak sekcji daje 404.

P0-02: Postęp ucznia sterowany przez klienta
Problem:
Klient nie może sam deklarować courseId, sectionId, statusu, quizCompleted, taskCheckedByAi itd.

Wymagania:
1. Endpoint /progress ma przyjmować tylko bezpieczne dane:
   - topicId,
   - currentElementType: "video" | "quiz" | "task",
   - ewentualnie videoCompleted przy realnym zdarzeniu video.
2. Backend musi sam wyliczać:
   - sectionId,
   - courseId,
   - relację topic -> section -> course.
3. Backend musi sprawdzić aktywny dostęp użytkownika do kursu.
4. Flagi:
   - quizCompleted ustawiaj tylko po prawdziwie zaliczonym quizie w routes/quizzes.ts,
   - taskCheckedByAi ustawiaj tylko po prawdziwym sprawdzeniu AI w routes/ai.ts,
   - videoCompleted ustawiaj tylko przez bezpieczne zdarzenie video.
5. Użyj zod schema.
6. Użyj upsert/onConflictDoUpdate w kontrolowany sposób.
7. Dodaj testy bezpieczeństwa:
   - user nie może zapisać postępu do kursu bez dostępu,
   - user nie może podmienić courseId/sectionId,
   - user nie może oznaczyć quiz/task jako ukończone przez /progress,
   - nieistniejący topicId daje 404,
   - invalid body daje 400.

P0-03: Build/typecheck/test/runtime
Wymagania:
1. Dodaj w root package.json:
   - packageManager: "pnpm@9.15.9" albo stabilną wersję zgodną z lockfile,
   - skrypty: typecheck, build, test, test:all, lint jeżeli linter istnieje albo dodaj ESLint.
2. Komendy muszą przejść:
   - corepack prepare pnpm@9.15.9 --activate
   - pnpm install --frozen-lockfile
   - pnpm run typecheck
   - pnpm --filter @workspace/api-server run test
   - PORT=8080 BASE_PATH=/ pnpm --filter @workspace/physics-platform run build
   - pnpm --filter @workspace/api-server run build
   - docker compose down -v
   - docker compose up -d --build
   - docker compose logs --no-color api web db
3. Nie kończ pracy, jeśli któraś komenda nie przechodzi. Napraw błędy.

P1-02: Usuń hardcoded courseId=1
Wymagania:
1. Żaden flow ucznia, płatności ani admin nie może zakładać courseId=1.
2. GET /topics/:topicId musi zwracać:
   - topicId,
   - sectionId,
   - courseId,
   - nextTopicId,
   - previousTopicId, jeśli możliwe,
   - statusy/progress.
3. Frontend topic-detail.tsx musi używać topic.courseId i topic.sectionId z API.
4. payments/create musi przyjmować courseId, walidować kurs jako aktywny/published i liczyć cenę po stronie backendu.
5. Admin grant access ma mieć wybór kursu, nie default 1.
6. Seed może tworzyć domyślny kurs, ale aplikacja nie może zależeć od ID = 1.

P1-03: Płatności mock/dev i success page
Wymagania:
1. Mock płatności tylko w dev/test albo za jawnie ustawioną flagą.
2. /payment/success ma obsłużyć:
   - normalny powrót z Przelewy24,
   - mock=true&paymentId=... w dev.
3. Jeśli mock=true:
   - wywołaj POST /payments/mock-complete/:paymentId tylko raz,
   - odśwież /me i accessGrants,
   - pokaż jasny stan loading/success/error.
4. Dodaj test e2e:
   rejestracja -> zakup mock -> success -> dashboard pokazuje dostęp.

P1-04: Bezpieczny returnUrl w płatnościach
Wymagania:
1. Backend nie może ufać returnUrl z klienta.
2. Domyślnie używaj config.appUrl + "/payment/success".
3. Jeśli przyjmujesz returnUrl:
   - parsuj przez new URL,
   - sprawdzaj origin === new URL(config.appUrl).origin,
   - w przeciwnym razie fallback.
4. Dodaj testy:
   - returnUrl z obcej domeny odrzucony/fallback,
   - poprawny returnUrl zaakceptowany,
   - niepoprawny string fallback.

P1-05: Provider IDs w płatnościach
Problem:
providerPaymentId nie może mieszać sessionId i orderId.

Wymagania:
1. Dodaj migrację:
   - providerSessionId,
   - providerOrderId,
   - providerPaymentId jeśli nadal potrzebny, ale semantycznie czysty.
2. Zaktualizuj routes/payments.ts.
3. Webhook P24 musi być idempotentny.
4. Dodaj testy:
   - zły sign,
   - zła kwota,
   - poprawny sign,
   - verify success,
   - ponowny webhook nie duplikuje dostępu,
   - dostęp powstaje tylko po potwierdzonej płatności.

P1-06: Quizy wyłącznie A/B/C/D
Wymagania:
1. Frontend:
   - LETTERS = ["A", "B", "C", "D"],
   - nie da się dodać 5 odpowiedzi,
   - UI pokazuje dokładnie 4 odpowiedzi.
2. Backend:
   - answerLabel tylko A-D,
   - każde pytanie musi mieć dokładnie 4 odpowiedzi,
   - dokładnie jedna odpowiedź correct,
   - quiz nie może być opublikowany, jeśli warunki nie są spełnione.
3. Quiz submit:
   - wymaga odpowiedzi na wszystkie pytania,
   - scoring server-side,
   - correctAnswerId zwracaj dopiero po submit,
   - progress upsert po zaliczeniu.
4. Testy:
   - label E odrzucony,
   - 5 odpowiedź odrzucona,
   - brak correct odrzucony,
   - dwie correct odrzucone,
   - submit niekompletny odrzucony.

P1-07: Auth produkcyjny
Wymagania:
1. Przenieś auth z localStorage JWT na HttpOnly, Secure, SameSite cookie, jeśli architektura pozwala.
2. Jeśli pełne cookie auth wymaga większej przebudowy, wdroż co najmniej:
   - krótkie access tokeny,
   - refresh token lub session rotation,
   - tokenVersion/sessionVersion w DB,
   - logout invaliduje sesję,
   - ban usera blokuje aktywne sesje.
3. Walidacja backend:
   - email normalizowany,
   - password min 8,
   - litera + cyfra,
   - max length,
   - zod schema.
4. Frontend i backend mają identyczne reguły walidacji.
5. Nie trzymaj sekretów w frontendzie.

P1-08: AI/Gemini
Wymagania:
1. Gemini API key tylko backend.
2. Prompt AI tylko backend.
3. taskId konwertuj i waliduj jako number.
4. Waliduj:
   - typ obrazu,
   - limit 5 MB lub zgodny z audytem,
   - user access do kursu/zadania,
   - dzienny limit AI.
5. Po sprawdzeniu AI:
   - zapisz ai_checks,
   - ustaw progress przez kontrolowany upsert,
   - nie ufaj klientowi.
6. UI:
   - feedback AI jako czytelna karta:
     “Co dobrze”, “Co poprawić”, “Następny krok”.
   - na mobile jako accordion/card.
7. Dodaj testy access/validation/rate limit.

P1-09: Admin API i panel
Wymagania:
1. Wszystkie endpointy admin:
   - requireAuth,
   - requireAdmin,
   - zod validation,
   - transakcje przy operacjach wieloetapowych,
   - audit log po poprawnym wykonaniu operacji.
2. Guardy:
   - admin nie może przypadkiem zbanować/usunąć sam siebie,
   - nie można odebrać ostatniemu adminowi roli admina,
   - waliduj courseId.
3. Mobile admin:
   - tabele zamień na card layout na 320-430 px,
   - sticky actions,
   - czytelne formularze,
   - inline errors.
4. Dodaj testy admin:
   - non-admin 403,
   - invalid body 400,
   - self-ban blocked,
   - grant access valid,
   - quiz/task/video CRUD valid.
