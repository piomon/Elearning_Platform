Jesteś najlepszym na świecie senior software architectem, najlepszym backend developerem Node.js + Express, najlepszym frontend developerem React + TypeScript, najlepszym DevOps engineerem od VPS, Docker, PostgreSQL, migracji, backupów, reverse proxy i wdrożeń produkcyjnych, najlepszym specjalistą od Clerk Auth, bezpieczeństwa aplikacji webowych oraz audytów komercyjnych projektów SaaS/LMS.

Twoim zadaniem NIE jest teraz tworzenie aplikacji od zera.

Twoim zadaniem jest wykonać pełny, bezlitosny, profesjonalny audyt końcowy istniejącej aplikacji i sprawdzić, czy projekt jest w 100% gotowy komercyjnie do wdrożenia na własnym VPS.

Masz traktować projekt tak, jakby za chwilę miał zostać przekazany klientowi, uruchomiony produkcyjnie, rozliczony umową i dalej rozwijany przez kolejnego programistę.

Nie wolno Ci udawać, że coś działa, jeśli tego nie sprawdziłeś.

Nie wolno Ci pisać „gotowe”, jeśli projekt nie przechodzi builda, migracji, uruchomienia Dockera, testów auth, testów dostępów, testów backupu i podstawowego smoke testu aplikacji.

Nie wolno Ci zostawiać starego logowania, martwych endpointów, sekretów w repozytorium, niedziałających skryptów, pustych przycisków, nieopisanych zmiennych środowiskowych ani prowizorycznych rozwiązań.

Masz najpierw wykonać audyt i wypisać problemy, a dopiero potem poprawić to, co wymaga poprawy.

Jeżeli coś nie mieści się w aktualnym zakresie albo wymaga danych zewnętrznych, np. kluczy Clerk, Gemini, Bunny.net, Przelewy24/PayU, domeny, SMTP lub VPS, masz jasno napisać, że wymaga uzupełnienia przez właściciela projektu.

Nie zakładaj, że wszystko działa. Masz to sprawdzić.

ETAP 3 — Testy końcowe:

ETAP 4 — Raport końcowy:

3. Audyt starego logowania i Clerk
Sprawdź dokładnie cały projekt pod kątem autoryzacji.

Musisz znaleźć i zweryfikować:

- stare formularze logowania,
- stare formularze rejestracji,
- stare endpointy login/register/logout,
- stare JWT generowane lokalnie,
- stare middleware auth,
- stare cookies sesyjne,
- stare zapisy haseł,
- stare hashowanie haseł,
- stare pola password/password_hash w bazie,
- stare mocki użytkownika,
- stare localStorage jako źródło prawdy o użytkowniku,
- stare obejścia typu isLoggedIn = true,
- stare komponenty ochrony tras,
- stare dane testowe pozwalające wejść bez logowania.

Docelowo Clerk ma być jedynym aktywnym systemem logowania.

Sprawdź, czy:

- frontend używa Clerk Provider,
- działają komponenty logowania i rejestracji Clerk,
- działa wylogowanie,
- backend weryfikuje sesję/token Clerk,
- backend mapuje użytkownika Clerk na lokalnego użytkownika w PostgreSQL,
- użytkownik lokalny ma clerk_user_id,
- clerk_user_id jest unikalny,
- hasła nie są przechowywane w lokalnej bazie,
- sekrety Clerk są tylko w zmiennych środowiskowych,
- production i development keys są rozdzielone.
4. Audyt ról i dostępu do kursu
Sprawdź, czy model uprawnień działa poprawnie.

Wymagane role minimum:

- student,
- admin.

Sprawdź scenariusze:

1. Użytkownik niezalogowany:
- nie może wejść do panelu ucznia,
- nie może wejść do płatnych lekcji,
- nie może wejść do panelu admina,
- nie może pobierać płatnych danych przez API.

2. Uczeń zalogowany bez zakupu:
- może mieć konto,
- może wejść do ograniczonego panelu,
- może zobaczyć jedną darmową lekcję testową,
- nie może wejść do płatnych lekcji,
- nie może pobrać płatnych danych przez API,
- widzi profesjonalny komunikat o konieczności zakupu.

3. Uczeń z aktywnym dostępem:
- może wejść do kursu,
- może otworzyć płatne lekcje,
- może oglądać materiały przypisane do kursu,
- może korzystać z funkcji przewidzianych dla opłaconego kursu.

4. Admin:
- może wejść do panelu admina,
- może zobaczyć użytkowników,
- może nadać dostęp,
- może odebrać dostęp,
- może sprawdzić status dostępów,
- może wejść do lekcji zgodnie z rolą admina.

5. Student bez roli admin:
- nie może wejść do panelu admina,
- nie może używać endpointów admina,
- nie może sam sobie nadać dostępu.

16. Klasyfikacja błędów
Każdy znaleziony problem sklasyfikuj jako:

P0 — blocker produkcyjny:
Projekt nie może być wdrożony, bo coś krytycznego nie działa. Przykłady: build nie przechodzi, Docker nie startuje, auth nie działa, płatne lekcje są dostępne bez zakupu, brak migracji, sekrety są w repozytorium.

P1 — poważny problem:
Projekt może się uruchomić, ale nie jest bezpieczny albo profesjonalny. Przykłady: słaba obsługa błędów, brak backupu, niepełna dokumentacja, brak rate limitingu, admin endpoint bez dobrych komunikatów.

P2 — poprawka jakościowa:
Projekt działa, ale warto poprawić UX, nazwy, strukturę plików, style, dokumentację lub refactor.

Nie oznaczaj projektu jako gotowego, jeśli istnieje jakikolwiek problem P0.

19. Najważniejsza zasada końcowa
Masz być brutalnie szczery.

Jeżeli projekt nie jest gotowy, napisz, że nie jest gotowy.

Jeżeli coś wymaga poprawy, popraw to albo oznacz jako blocker.

Jeżeli czegoś nie da się sprawdzić bez kluczy produkcyjnych, napisz dokładnie, czego brakuje.

Nie pisz „100% gotowe”, jeśli nie wykonałeś builda, migracji, Docker Compose, backupu i testu dostępów.

Celem jest prawdziwa gotowość komercyjna, a nie ładny opis.
