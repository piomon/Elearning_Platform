## 1. Aktualny stan aplikacji z załączonego ZIP

Repozytorium jest monorepo aplikacji edukacyjnej.

Stack:

* frontend: React + TypeScript + Vite + Tailwind,
* backend: Node.js + Express + TypeScript,
* baza danych: PostgreSQL + Drizzle ORM,
* API: OpenAPI + wygenerowany klient React Query,
* AI: Google Gemini API po stronie backendu,
* wideo: Bunny.net Stream,
* tablica: Excalidraw,
* płatności obecnie w kodzie: Przelewy24/P24,
* docelowo teraz: Paynow Sandbox i później Paynow produkcja,
* wdrożenie: Docker / Docker Compose / VPS / Nginx / Traefik,
* testy: Vitest backend, typecheck, build, docelowo także E2E/Playwright.

Główne katalogi:

* `artifacts/api-server` — backend Express,
* `artifacts/physics-platform` — frontend React,
* `lib/db` — schema i migracje Drizzle,
* `lib/api-spec` — OpenAPI,
* `lib/api-client-react` — wygenerowany klient,
* `lib/api-zod` — wygenerowane typy/schematy,
* `scripts` — seed i dane kursu,
* `docker` — Dockerfile API i web,
* `deploy` — skrypty wdrożeniowe,
* `attached_assets` — audyt, prompty, kurs, quizy, PNG, materiały pomocnicze.

Według plików kursu aplikacja zawiera:

* 3 główne działy kursu,
* 21 lekcji,
* 14 quizów,
* 34 obrazki PNG,
* mapę filmów Bunny.net,
* 56 filmów w Bunny według mapy `bunny-videos.json`,
* strukturę kursu i quizów w `scripts/src/course-data.ts`.

To wszystko ma zostać sprawdzone i wykorzystane, a nie pominięte.

## 8. Testy Paynow Sandbox / BLIK

Przetestuj proces płatności BLIK w Paynow Sandbox.

Podczas wyboru BLIK użyj kodów testowych:

* `111 111` — powinien skutkować sukcesem transakcji,
* `333 333` — powinien wymusić błąd autoryzacji.

Scenariusz sukcesu `111 111`:

1. Użytkownik klika zakup.
2. Backend tworzy płatność Paynow.
3. Użytkownik trafia do Paynow Sandbox.
4. Użytkownik wybiera BLIK.
5. Użytkownik wpisuje `111 111`.
6. Paynow potwierdza sukces.
7. Webhook trafia do backendu.
8. Backend weryfikuje webhook.
9. Backend aktualizuje status płatności.
10. Backend nadaje dostęp do kursu.
11. Uczeń po powrocie widzi odblokowany kurs/lekcje.
12. Nie jest wymagana ręczna interwencja administratora.

Scenariusz błędu `333 333`:

1. Użytkownik klika zakup.
2. Backend tworzy płatność Paynow.
3. Użytkownik wybiera BLIK.
4. Użytkownik wpisuje `333 333`.
5. Paynow zwraca błąd.
6. Backend nie nadaje dostępu.
7. Status płatności zapisuje się jako nieudany/odrzucony/anulowany.
8. Użytkownik dostaje czytelny komunikat.
9. Użytkownik może ponowić płatność.
10. Kurs pozostaje zablokowany.

## 12. Blokery bezpieczeństwa i produkcji

Sprawdź i potwierdź, że nadal zamknięte są wszystkie wcześniejsze blokery P0/P1 z audytu i raportu MVP:
