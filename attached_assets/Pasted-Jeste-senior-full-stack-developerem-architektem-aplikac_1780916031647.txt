Jesteś senior full-stack developerem, architektem aplikacji SaaS/LMS, specjalistą od bezpieczeństwa aplikacji webowych, UX/UI dla platform edukacyjnych oraz DevOps Docker/VPS. Twoim zadaniem jest dokończyć, naprawić, zabezpieczyć i wizualnie dopracować MVP platformy edukacyjnej z modułem AI dla kursu fizyki.

Masz pracować na istniejącym repozytorium aplikacji:
- frontend: React + TypeScript + Vite + Tailwind,
- backend: Node.js + Express + TypeScript,
- baza danych: PostgreSQL + Drizzle,
- API: OpenAPI + wygenerowany klient React Query,
- moduły: panel ucznia, panel admina, quizy, Excalidraw, Google Gemini AI, Bunny.net/wideo, Przelewy24/PayU, kontakt, postęp nauki, Docker/VPS.

Twoim celem nie jest tylko kosmetyczna poprawka. Masz doprowadzić projekt do realnego, bezpiecznego, odbieralnego MVP produkcyjnego.

WAŻNE:
- Nie traktuj obecnego kodu jako gotowego.
- Najpierw napraw blokery P0 z audytu, potem P1, potem UX/UI i kosmetykę.
- Nie ukrywaj problemów frontendowymi obejściami.
- Wszystkie płatne zasoby muszą być chronione backendowo.
- Wszystkie kontrakty API muszą być zgodne między backendem, OpenAPI i wygenerowanym klientem.
- Nie zostawiaj mocków, demo endpointów ani sandboxowych płatności aktywnych w produkcji.
- Nie kopiuj 1:1 wyglądu LearnWorlds. Zainspiruj się stylem: nowoczesny SaaS/LMS, jasny layout, mocny hero, błękitne gradienty, karty funkcji, sekcje zaufania, sprzedażowy landing, czysty dashboard. Projekt ma być autorski.

ETAP 1 — Stabilizacja konfiguracji i bezpieczeństwa startowego

1. Usuń fallback JWT:
   - znajdź miejsce typu:
     JWT_SECRET ?? SESSION_SECRET ?? "dev_secret_change_me"
   - usuń stały fallback,
   - aplikacja ma przerywać start, jeśli JWT_SECRET nie istnieje albo ma mniej niż 32 znaki.

2. Dodaj centralny plik konfiguracji env, np.:
   artifacts/api-server/src/config/env.ts

   Funkcja:
   requiredEnv(name: string, minLength = 1): string

   Ma rzucać błąd przy braku wymaganej zmiennej.

3. Dodaj wymagane env:
   - JWT_SECRET,
   - DATABASE_URL,
   - APP_URL,
   - API_URL,
   - ALLOWED_ORIGINS,
   - GEMINI_API_KEY,
   - P24_ENV,
   - P24_MERCHANT_ID,
   - P24_POS_ID,
   - P24_API_KEY,
   - P24_CRC,
   - COURSE_PRICE_GROSZ,
   - BUNNY_LIBRARY_ID,
   - BUNNY_API_KEY,
   - BUNNY_EMBED_BASE_URL,
   - SMTP_HOST,
   - SMTP_PORT,
   - SMTP_USER,
   - SMTP_PASS,
   - CONTACT_FROM_EMAIL.

4. CORS:
   - nie używaj cors() otwartego dla każdego origin,
   - użyj ALLOWED_ORIGINS z .env,
   - produkcyjnie dopuszczaj tylko domenę aplikacji.

5. Body limit:
   - ustaw express.json({ limit: "5mb" }),
   - dla AI image endpoint dodaj osobną walidację i limit.

6. Rate limit:
   - /auth/login,
   - /auth/register,
   - /contact,
   - /ai/check,
   - /payments/create,
   - webhooki ostrożnie, aby nie zablokować operatora.

ETAP 2 — Poprawa kontroli dostępu

Obecny requireAccess sprawdza tylko dowolny aktywny grant użytkownika. To jest błędne.

Zaimplementuj:

1. assertUserHasCourseAccess(userId, courseId):
   - sprawdza:
     access_grants.user_id = userId,
     access_grants.course_id = courseId,
     status = "active",
     validFrom <= now,
     validTo IS NULL OR validTo > now.

2. getCourseIdByTopicId(topicId):
   - topic → section → course.

3. getCourseIdByQuizId(quizId):
   - quiz → topic → section → course.

4. getCourseIdByTaskId(taskId):
   - task → topic → section → course.

5. Middleware/funkcje:
   - requireCourseAccess(courseId),
   - requireTopicAccess(topicId),
   - requireQuizAccess(quizId),
   - requireTaskAccess(taskId).

6. Zastosuj access check w:
   - GET /topics/:topicId,
   - GET /quizzes/:quizId,
   - POST /quizzes/:quizId/attempts,
   - POST /ai/check,
   - wszystkich płatnych endpointach materiałów.

7. /auth/me:
   - nie zwracaj prostego hasAccess na podstawie dowolnego grantu,
   - zwracaj accessGrants z courseId, status, validFrom, validTo,
   - albo licz hasAccess względem głównego kursu, ale z poprawnym warunkiem courseId + validTo.

ETAP 3 — Naprawa quizów

To jest krytyczne.

1. Usuń isCorrect ze wszystkich endpointów ucznia:
   - GET /topics/:topicId,
   - GET /quizzes/:quizId.

2. Stwórz osobne DTO:
   - StudentQuizAnswer: id, questionId, answerLabel, answerText,
   - AdminQuizAnswer: id, questionId, answerLabel, answerText, isCorrect.

3. GET quiz dla ucznia:
   - wymaga requireQuizAccess,
   - zwraca pytania i odpowiedzi bez isCorrect.

4. POST submit quiz:
   - wymaga requireQuizAccess,
   - pobiera pytania tylko dla danego quizId,
   - pobiera odpowiedzi tylko dla tych pytań,
   - buduje mapy:
     questionId → correctAnswerId,
     answerId → questionId,
   - waliduje:
     czy questionId należy do quizu,
     czy selectedAnswerId istnieje,
     czy selectedAnswerId należy do tego questionId,
     czy użytkownik nie wysłał odpowiedzi z innego quizu,
   - odrzuca request 400 przy obcych ID,
   - liczy wynik po stronie serwera,
   - zapisuje attempt i attempt_answers w transakcji,
   - dopiero po submit zwraca:
     score,
     totalQuestions,
     percentage,
     answersReview: questionId, selectedAnswerId, correctAnswerId, isCorrect.

5. Dodaj testy:
   - GET quiz nie zwraca isCorrect,
   - submit z answerId z innego quizu zwraca 400,
   - uczeń bez access dostaje 403,
   - grant wygasły validTo < now daje 403.

ETAP 4 — Naprawa postępu ucznia

Obecnie aktualizacja postępu filtruje tylko po userId, przez co może aktualizować wiele rekordów.

   - filtruj po userId + topicId,

3. Dodaj onConflictDoUpdate:

     currentElementType,
     videoCompleted,
     quizCompleted,
     taskStarted,
     taskCheckedByAi,
     updatedAt.

4. Dashboard:
   - procent postępu ma wynikać z realnych danych,

     totalTopics,
     completedVideos,
     completedQuizzes,
     completedTasks,
     overallPercent,
     lastTopic,

5. Continue:

ETAP 5 — Płatności Przelewy24/PayU

Nie wolno zostawić mocków jako produkcyjnego flow.

1. Cena:
   - usuń rozjazd 199 zł vs 299 zł,
   - COURSE_PRICE_GROSZ ma być w backendzie/configu/DB,
   - frontend pobiera cenę z API,
   - formatowanie PLN przez Intl.NumberFormat.

2. Frontend:
   - przycisk „Kup dostęp teraz” nie może prowadzić tylko do /register,
   - jeśli użytkownik niezalogowany:
     zapisz intencję zakupu,
     po rejestracji/logowaniu wróć do create payment,
   - jeśli zalogowany:
     POST /payments/create z courseId,
     redirect do redirectUrl operatora.

3. Backend:
   - POST /payments/create:
     wymaga auth,
     tworzy pending payment,
     generuje sessionId,
     rejestruje transakcję w P24/PayU,
     zwraca redirectUrl.

4. P24_ENV:
   - sandbox → URL sandbox,
   - production → URL produkcyjny,
   - produkcja nigdy nie może używać sandboxa.

5. Webhook/status:
   - po otrzymaniu webhooka nie nadawaj dostępu od razu,
   - wykonaj verify transakcji u operatora,
   - porównaj:
     sessionId,
     amount,
     currency,
     status,
     providerPaymentId,
   - dopiero wtedy:
     payment.status = completed,
     twórz access_grant.

6. Idempotencja:
   - duplikat webhooka nie może tworzyć drugiego grantu,
   - dodaj unique/indexy dla payment/session/providerPaymentId,
   - access_grants powinien mieć kontrolę jednego aktywnego grantu per user/course albo jawnie obsługiwane przedłużenia.

7. mock-complete:
   - rejestruj endpoint tylko w NODE_ENV=development/test,
   - w production endpoint ma nie istnieć lub zwracać 404.

8. Payment success page:
   - nie aktywuje dostępu,
   - pokazuje status,
   - odpytuje backend,
   - odświeża /auth/me.

ETAP 14 — Testy obowiązkowe

Dodaj testy minimalne:

API/auth:
- register waliduje hasło,
- login ma rate limit,
- banned user nie może się zalogować,
- /auth/me zwraca poprawne accessGrants.

API/access:
- brak grantu → 403 dla topic/quiz/AI,
- grant wygasły → 403,
- grant do innego courseId → 403,
- aktywny grant do kursu → 200.

API/quiz:
- GET quiz nie zwraca isCorrect,
- submit obcego answerId → 400,
- submit niepełny → 400 albo zgodnie z ustaloną regułą,
- wynik liczony po stronie serwera,
- progress aktualizuje tylko userId + topicId.

API/payments:
- create payment tworzy pending,
- webhook z błędną kwotą nie daje dostępu,
- webhook poprawny daje completed i access_grant,
- duplikat webhooka nie tworzy drugiego grantu,
- mock-complete nie działa w production.

API/AI:
- brak dostępu → 403,
- za duży obraz → 413/400,
- zły MIME → 400,
- brak GEMINI_API_KEY w prod → 503,
- development demo tylko w dev.

E2E:
- rejestracja → płatność sandbox → dostęp → dashboard → lekcja,
- admin dodaje temat z wideo, quizem i zadaniem → uczeń widzi temat,
- formularz kontaktowy → wiadomość widoczna w adminie,
- ban usera → user nie może korzystać.

Visual QA:
